# 月度绩效书写指南

本指南包含详细的写法模板和完整示例，供改写时参考。

## 篇幅参考

| 部分 | 建议字数 | 说明 |
|------|----------|------|
| 主要工作快报 | 每条 50-80 字 | 3-5 条，核心项目排前面 |
| 经验 | 每条 80-120 字 | 1-2 条，宁精勿滥 |
| 不足 | 每条 40-60 字 | 2-3 条，简洁有力 |
| 落实情况 | 每条 30-50 字 | 逐条回应，体现闭环 |

以上为参考范围，根据用户实际工作量灵活调整。整体以充实、不单薄为原则。

---

## 主要工作快报

### 写法要点

每条工作体现三要素：**动作 + 结果 + 价值**

升华结构：
```text
周报原文：做了什么
改写后：围绕【目标/场景】，完成【动作/交付物】，达成【结果/影响】，为【后续/更远目标】奠定基础。
```

| 维度 | 示例 |
|------|------|
| 动作 | 完成 XX 系统全链路开发 |
| 结果 | 接口响应时间降低 60% |
| 价值 | 支撑业务方 YY 场景上线，预计覆盖 N 万用户 |

### 量化策略

从周报中提取可用数据，没有精确数字时用范围描述：
- 数量级："完成 10+ 个需求"、"修复多个线上缺陷"
- 覆盖面："覆盖全部核心场景"、"支持 3 个业务方"
- 对比提升："较之前有明显提升"、"性能指标达到预期目标"

### 好与不好的对比

```text
好：完成支付模块全链路开发与联调，修复 3 个阻塞性问题，
    按期完成里程碑验收，支撑业务侧按时上线

不好：做了支付模块的开发和联调
```

### 核心要点

- **结果前置**：先说达成了什么，再说过程
- **体现业务价值**：关联业务目标或用户影响
- **按重要性排序**：核心项目排前面
- **合并同类项**：跨周重复出现的工作合并为一条，体现持续性推进

## 经验写法（深度分析）

经验是绩效的加分项，要写得有深度、有成长感。

### 写法框架

**场景 + 认知 + 方法论沉淀**

1. **从具体工作中提炼**：不是空泛的"学到了很多"，而是绑定具体场景
2. **体现思考深度**：不只是"做了什么"，而是"为什么这样做更好"
3. **展示方法论沉淀**：把个人经验上升到可复用的方法或认知
4. **体现能力成长**：问题定位、风险识别、跨团队协作、架构设计、交付质量

### 经验提炼问题

从周报的工作内容中思考以下问题来提炼经验：

1. 这个月遇到的**最难的问题**是什么？怎么解决的？
2. 做了什么**关键判断或取舍**？为什么这样选？
3. 从重复出现的问题中总结出了什么**规律或方法**？
4. 在哪些**能力维度上有明显提升**？
5. 哪些经验是**可以复用到后续工作**中的？

### 经验方向参考

根据实际工作内容选择最贴合的方向：
- **技术判断力**：方案选型时的决策过程和理由
- **问题定位能力**：复杂问题的排查思路和方法
- **推动力**：跨团队/跨角色推动事情落地的经验
- **流程改进**：发现流程问题并主动优化的案例
- **风险意识**：提前识别风险并前置处理的实例
- **能力跃迁**：从执行某个任务到独立设计某个方案的成长

### 好与不好的对比

```text
# 好的经验写法
在推进 XX 项目的过程中，发现 YY 环节是主要瓶颈。通过分析链路耗时，
采用 ZZ 方案优化后，处理效率提升 3 倍。这次经历让我更深刻地认识到：
在复杂项目中，先定位瓶颈再优化比全面铺开更高效。后续已将这一思路
整理成架构评审 checklist，供团队参考。

# 不好的经验写法
本月学到了很多，对项目有了更深入的理解。
```

## 不足写法（积极美化）

不足的目标是让领导觉得你**有自我认知、有进取心、已经在行动**，而不是觉得你有问题。

### 核心策略

**把"不足"包装为"追求更高标准时的发现"**

每条不足包含三个要素：
1. **场景/背景**：在什么情况下发现的
2. **改进方向**：不是问题本身，而是要提升的方向
3. **已有行动**：已经做了什么或计划做什么

### 美化策略

**策略一：高标准视角** — 不是"做得不好"，而是"在追求更高标准时发现可以更好"

| 原始表达 | 美化后 |
|---------|--------|
| 代码质量不够好 | 在追求更高交付质量的过程中，发现代码审查环节还有进一步精细化的空间 |
| 进度有延迟 | 在保证质量的前提下，发现项目管理颗粒度还可以更细，以提升交付的确定性 |
| 沟通不够及时 | 随着协作范围扩大，意识到在信息同步的及时性和透明度上还有提升空间 |

**策略二：主动发现** — 不是被动暴露问题，而是主动识别改进空间

```text
在复盘本月项目交付过程时，主动梳理了 XX 环节的效率瓶颈，
发现通过引入 YY 机制可以进一步提升。已开始调研相关方案，
计划下月在 ZZ 项目中试点验证。
```

**策略三：成长阶段** — 不足是成长中的阶段性状态，不是能力缺陷

```text
随着本月承担的职责从 XX 拓展到 YY，对 ZZ 能力提出了更高要求。
目前在 WW 方面已开始系统学习，预计下月能在实际项目中实践验证。
```

### 不足转化为正面信号

- "做得不够好" → "对质量标准有更高追求"
- "沟通有问题" → "在复杂协作场景中探索更高效的沟通方式"
- "效率偏低" → "在探索更系统化的工作方法"
- "知识有盲区" → "持续学习，拓展能力边界"

### 不足的禁忌

- 不要写"还需要继续努力"这类空话——必须有具体改进动作
- 不要写真正的能力缺陷或重大失误——那是减分项
- 不要写太多条——2-3 条即可，多了显得问题多
- 不要写和别人对比的不足——只谈自己的提升空间

## 落实情况写法

**核心原则：用本月实际工作事项来回应上月领导反馈的建议，而非凭空编写。**

处理步骤：
1. 读取上月领导反馈的建议（每条）
2. 在本月提取到的工作事项中，找到与该建议相关的实际工作
3. 用该实际工作成果来体现建议的落实情况
4. 如本月工作中确实没有涉及某条建议，如实说明进展和后续计划

### 状态分类

**已完成**：
```text
1. 建议提升代码审查质量：已完成。引入 checklist 机制，
   本月审查覆盖率达 100%，线上缺陷数较上月下降。
```

**推进中**：
```text
2. 建议完善技术文档：大部分完成。核心模块文档已补齐（4/5），
   剩余部分已排入下月计划，预计中旬完成。
```

**未完成（美化原因）**：
```text
3. 建议搭建自动化测试框架：因本月优先支撑 XX 紧急上线任务，
   暂缓推进。技术方案已完成预研，下月正式启动。
```

**注意**：落实情况中提到的每项工作，必须能在本月"主要工作快报"中找到对应依据，不能凭空编造。

## 完整输出示例

以下是一份完整的月度绩效自评样例，展示各部分的篇幅、风格和衔接方式。

```markdown
**工作回顾**
内容可以包括：

（1）主要工作快报

- 完成财务结算系统重构全链路开发，涵盖对账、结算、清分三大核心模块，
  通过引入异步处理机制，将结算耗时从分钟级优化至秒级，
  支撑业务侧顺利完成首次月度结算，结算准确率达 100%。

- 完成三方支付渠道（微信、支付宝）接入方案设计与联调，
  解决签名校验、回调幂等等 3 个关键技术问题，
  按期通过联调验收，为 Q3 支付能力扩展奠定基础。

- 修复线上支付回调偶发失败问题，通过日志链路追踪定位到网络超时重试机制缺陷，
  优化后回调成功率从 95% 提升至 99.8%，线上客诉量明显下降。

- 配合测试团队完成财务优化模块的全面回归测试，
  梳理测试用例 80+ 条，主动发现并修复 2 个边界场景缺陷，
  保障模块高质量上线。

（2）经验和不足

经验：
- 在推进三方支付渠道接入时，面对多家渠道接口规范不一致的情况，
  通过抽象统一适配层，将渠道差异屏蔽在接入层，后续新增渠道只需实现适配接口即可。
  这个经验让我更深刻地认识到：在涉及多外部系统对接的场景中，
  提前做好架构抽象比逐个对接更高效，后续已将这一思路沉淀为团队对接规范。

不足：
- 在追求更快交付节奏的过程中，发现代码审查环节还有进一步精细化的空间。
  已在月末引入 review checklist 机制，下月重点验证效果。
- 随着本月承担的职责范围扩展到三方业务，对业务域的理解深度还需持续加强。
  已开始系统梳理三方业务流程，计划下月完成核心流程文档沉淀。

**上月绩效提升建议的落实情况**

1. 建议配合测试顺利完成财务优化的测试工作：已完成。本月主动配合测试团队
   完成全面回归测试，梳理测试用例 80+ 条，保障模块高质量上线。

2. 建议认真梳理三方业务需求并调研：已完成。本月完成三方支付渠道接入方案设计，
   调研了微信、支付宝等渠道接口规范，抽象统一适配层，按期通过联调验收。
```
