# Monthly Performance Writer

把每周周报内容提取、聚合、升华成月度绩效自评。经验要写出深度和成长感，不足要包装成积极进取的姿态。目标是让领导看到你的产出和潜力。

## 工作流程

### 阶段一：采集周报内容

用户通过以下任一方式提供本月周报：

1. **直接粘贴**：把每周周报内容粘贴到对话中
2. **提供文件路径**：指向保存周报的文件
3. **口头描述**：用户口述本月做了什么，skill 整理

通过 `AskUserQuestion` 询问用户选择哪种方式提供内容：

| 问题 | header | 选项 |
|------|--------|------|
| 请选择周报内容提供方式？ | 提供方式 | 直接粘贴周报内容（Other 输入）/ 提供文件路径（Other 输入）/ 口头描述（Other 输入） |

如果用户消息中已经包含了周报内容或明确说了"帮我写月度绩效，以下是本月周报..."，直接使用，跳过交互。

### 阶段二：确认月份

如果用户没有明确月份，通过 `AskUserQuestion` 确认：

| 问题 | header | 选项 |
|------|--------|------|
| 绩效对应的月份？ | 月份 | 本月 / 上月 / Other 输入 |

### 阶段三：加载上月改进建议

1. 执行 `git config user.name` 获取用户名。
2. 检查 `references/{用户名}-last-month.md` 是否存在。
   - **存在** → 读取上月改进建议，用于"落实情况"部分。
   - **不存在** → 该部分由周报内容推导或用户补充。

### 阶段四：提取与聚合

从周报中提取工作内容，按以下规则处理：

**提取**：从每周周报中提取所有工作事项，去重合并。

**聚合**：将同一主题/项目的工作合并为一条，按重要性和影响力排序。

```text
# 原始（第1周周报）
- 完成支付模块接口开发
- 修复支付回调偶发失败问题

# 原始（第3周周报）
- 支付模块性能优化，响应时间降低 40%

# 合并后
- 支付模块：完成接口开发与回调问题修复，并通过性能优化将响应时间降低 40%
```

**归类到输出结构**：
- 主要工作快报 → 核心产出和成果
- 经验和不足 → 从工作中提炼（经验深度展开，不足积极包装）
- 落实情况 → 对比上月建议逐条回应

### 阶段五：确认事项

聚合完成后展示提取结果，让用户确认：

```markdown
## 本月提取到以下工作事项

**主要工作**
1. ...
2. ...
3. ...

**可提炼的经验**
- ...

**上月改进建议（{数量}条）**
1. ...

---
如有遗漏或需要补充，请告诉我。确认后我开始改写。
```

通过 `AskUserQuestion` 询问确认。

### 阶段六：改写生成月度绩效

确认后按"输出结构"和"改写规则"生成最终内容。

### 阶段七：保存本月改进建议

从"经验和不足"部分提取改进建议，写入 `references/{用户名}-last-month.md`，下月自动加载。格式：

```markdown
# 上月（{月份}）改进建议

1. 建议内容...
2. 建议内容...
```

## 改写规则

### 主要工作快报

从周报中提取的每条工作，按此结构升华：

```text
周报原文：做了什么
改写后：围绕【目标/场景】，完成【动作/交付物】，达成【结果/影响】，为【后续/更远目标】奠定基础。
```

要点：
- **结果前置**：先说达成了什么，再说过程
- **能量化就量化**：从周报中提取数字，没有数字时用"多个""全面""系统性"等词体现范围
- **体现业务价值**：关联业务目标或用户影响
- **按重要性排序**：核心项目排前面
- **合并同类项**：跨周重复出现的工作合并为一条，体现持续性推进

### 经验写法（深度分析）

经验是绩效的加分项，要写得有深度、有成长感：

**写法框架**：场景 + 认知 + 方法论沉淀

1. **从具体工作中提炼**：不是空泛的"学到了很多"，而是绑定具体场景
2. **体现思考深度**：不只是"做了什么"，而是"为什么这样做更好"
3. **展示方法论沉淀**：把个人经验上升到可复用的方法或认知
4. **体现能力成长**：问题定位、风险识别、跨团队协作、架构设计、交付质量

```text
# 好的经验写法
在推进 XX 项目的过程中，发现 YY 环节是主要瓶颈。通过引入 ZZ 方式，将处理效率提升 N 倍。
这个经验让我更深刻地认识到：在复杂项目中，先定位瓶颈再优化比全面铺开更高效。后续已将这一思路应用到 WW 项目中。

# 不好的经验写法
本月学到了很多，对项目有了更深入的理解。
```

**经验提炼方向**（根据周报内容选择最贴合的）：
- 技术方案选型的判断力和决策力
- 复杂问题的拆解和定位能力
- 跨团队/跨角色协作的推动力
- 流程或工具的改进和沉淀
- 风险识别和前置处理意识
- 从执行到方案设计的能力跃迁

### 不足写法（积极美化）

不足的目标是让领导觉得你**有自我认知、有进取心、已经在行动**，而不是觉得你有问题。

**核心策略：把"不足"包装为"追求更高标准时的发现"**

```text
# 直接说不足（不好）
本月代码审查不够仔细，漏了几个 bug。

# 美化后的不足（好）
在追求更快交付节奏的过程中，发现代码审查环节还有进一步精细化的空间。已在月末引入 checklist 机制，下月重点验证效果。
```

**美化套路**：

1. **高标准视角**：不是"做得不好"，而是"在追求更高标准时发现可以更好"
   - "在保证质量的基础上，发现还可以在 XX 方面进一步提升"
   - "随着项目复杂度提升，对 XX 能力提出了更高要求"

2. **主动发现**：不是被动暴露问题，而是主动识别改进空间
   - "在复盘 XX 项目时，主动梳理了可优化环节"
   - "通过对比行业实践，发现团队在 XX 方面还有成长空间"

3. **已有行动**：每个不足必须附带已经开始或计划中的改进动作
   - "已制定改进计划，下月落地验证"
   - "已开始学习 XX，计划在 YY 场景中实践"

4. **定位为成长**：不足是成长中的阶段性状态，不是能力缺陷
   - "随着承担的职责范围扩大，在 XX 方面需要持续提升"
   - "在从 XX 角色向 YY 角色转变的过程中，ZZ 能力是下一步重点"

**不足转化为正面信号**：
- "做得不够好" → "对质量标准有更高追求"
- "沟通有问题" → "在复杂协作场景中探索更高效的沟通方式"
- "效率偏低" → "在探索更系统化的工作方法"
- "知识有盲区" → "持续学习，拓展能力边界"

### 落实情况写法

每条上月建议逐条回应，体现闭环：

```text
1. 建议提升代码审查质量：已完成。引入 checklist 机制后，本月审查覆盖率达 100%，线上缺陷数较上月下降。
2. 建议加强技术文档沉淀：大部分完成。核心模块文档已补齐（4/5），剩余部分已排入下月计划。
3. 建议优化跨团队协作流程：持续推进中。已与相关团队对齐新流程，下月进入试行阶段。
```

## 输出结构

```markdown
**工作回顾*

（1）主要工作快报

- ...
- ...
- ...

（2）经验和不足

经验：
- ...

不足：
- ...

**上月绩效提升建议的落实情况*

1. ...
2. ...
3. ...
```

如果用户公司模板不同，沿用用户的标题和格式。

---

## 上月改进建议持久化

保存在 `references/{git用户名}-last-month.md`，每月改写完成后自动更新。

---

## 最终检查清单

- 主要工作快报覆盖了本月核心产出，内容丰富
- 经验部分有深度分析，体现了思考和方法论沉淀
- 不足部分包装积极，让领导感受到进取心和自我驱动力
- 落实情况每条有明确状态，未完成的写清后续计划
- 整体内容量充实，不单薄
- 没有编造不存在的工作或数据
- 语气自信但不夸张，专业且真实
