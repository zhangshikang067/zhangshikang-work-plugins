# OKR Weekly Report Writer

使用这个 skill，把零散的工作记录或 git commit log 改成清晰、可信、方便领导阅读的 OKR 周报。输出要让工作成果、推进进度、风险处理和后续动作一眼能看懂。

## 工作流程

1. **交互式采集参数**：技能触发后，立即使用 `AskUserQuestion` 弹出交互式选择，让用户指定日期范围、项目范围和补充说明。如果用户在触发时已经明确说了完整参数（如”扫描 operate 项目，上周的”），可以跳过交互直接执行。如果用户指定”手动模式”，则跳过 git 扫描，仅使用用户提供的内容。
2. 先识别周报背景：周期、OKR/KR 编号、阅读对象、用户角色、原始事项、已有数据、阻塞问题和后续计划。
3. 如果用户提供优秀范例，先提炼范例的结构、措辞和表达习惯，再进行改写，尽量贴近公司内部风格。
4. 如果存在 `references/my-2026-h1-okr.md`，读取该文件用于 KR 归类和措辞对齐；如果该文件不存在，继续按通用周报规则处理。
5. 如果缺少关键数据，不要编造数字。可以不写该数据，或在必要时用简短的”可补充数据”提示。
6. 按 OKR/KR 分组。每组不要只列任务，要写成”做了什么 + 结果/影响 + 当前状态/风险/下一步”。
7. 加入”思考与分享”部分，体现判断力：风险识别、取舍、复盘、学习和具体后续动作。
8. 语气保持客观、克制、专业。避免自夸、道歉式表达或流水账。

## Git Log 采集规则

### 交互式参数采集

技能触发后，先检查用户消息中是否已经包含明确的参数信息。如果用户已经说清了日期范围、项目名和补充内容，直接使用。否则，使用 `AskUserQuestion` 一次弹出以下问题让用户选择：

```
AskUserQuestion({
  questions: [
    {
      question: "选择要扫描的日期范围？",
      header: "日期范围",
      options: [
        { label: "近 14 天", description: "默认范围，扫描最近两周的提交" },
        { label: "本周", description: "本周一到今天" },
        { label: "上周", description: "上周一到上周五" }
      ],
      multiSelect: false
    },
    {
      question: "选择要扫描的项目范围？",
      header: "项目范围",
      options: [
        { label: "扫描全部项目", description: "扫描 REPO_ROOT 下所有含 .git 的项目" },
        { label: "指定项目", description: "选择 Other 输入项目名，多个用空格分隔，如 operate third-order" }
      ],
      multiSelect: false
    },
    {
      question: "是否有 git log 之外的补充工作内容？",
      header: "补充说明",
      options: [
        { label: "无补充", description: "仅使用 git commit log 作为工作输入" },
        { label: "有补充", description: "选择 Other 输入补充内容，如"参加了需求评审""做了 code review""帮前端联调了广告回传""本周主要在做技术调研"等" }
      ],
      multiSelect: false
    }
  ]
})
```

根据用户回答解析参数：
- **日期范围**："近 14 天" → `--since="14 days ago"`；"本周" → 计算本周一的日期作为 `--since`，`--until` 为明天；"上周" → 计算上周一到上周五的日期范围；"Other" → 用户输入的自定义日期，支持 `YYYY-MM-DD` 格式（可给一个日期作为起始，结束默认今天）或自然语言描述（如"5 月中旬"）。
- **项目范围**："扫描全部项目" → 不过滤；"Other" → 用户输入的项目名列表，空格分隔。
- **补充说明**："无补充" → 不合并额外内容；"Other" → 用户输入的文字，合并到最终工作输入中。

### 扫描命令

根据交互式采集的参数，构建并执行 bash 命令采集 commit：

```bash
AUTHOR=”张识康”
REPO_ROOT=”/home/zsk/code/sier”
SINCE=”<根据用户选择的日期范围计算>”
UNTIL=”<根据用户选择的日期范围计算>”
# 如果用户选择了指定项目，只扫描匹配的；否则扫描全部
PROJECTS=”<用户指定的项目列表，空则扫描全部>”

for group in ai backend frontend ops; do
  for dir in $REPO_ROOT/$group/*/; do
    [ -e “$dir/.git” ] || continue
    project=$(basename “$dir”)
    if [ -n “$PROJECTS” ]; then
      echo “$PROJECTS” | grep -qw “$project” || continue
    fi
    commits=$(git -C “$dir” log --author=”$AUTHOR” \
      --since=”$SINCE” --until=”$UNTIL” \
      --format=”%ad %s” --date=short --no-merges 2>/dev/null)
    [ -z “$commits” ] && continue
    echo “=== $group/$project ===”
    echo “$commits”
    echo “”
  done
done
```

### 过滤规则

采集到 commit 后，过滤掉以下低信息量提交，不纳入工作事项：

| 过滤模式 | 示例 | 说明 |
|----------|------|------|
| 单词无内容 | `stash`、`doc`、`version`、`test` | 只有标签没有描述 |
| 纯操作 | `误操作恢复`、`stash`、`stash` | 无实质工作 |
| 纯版本号 | `1.0.1`、`v2.3` | 版本打标 |
| 空泛动词 | `修改`、`更新`、`调整`（后无具体内容） | 无法判断做了什么 |

**保留**所有有实质工作描述的 commit，即使简短（如 `修复小红书订单解密问题`）。

### 聚合规则

将同一项目中**主题相关**的 commit 合并为一条工作事项：

```
# 原始（backend/operate）
c4129222 小红书平台归因
2f7e08ce 调试小红书广告归因回传
dfbe8ad0 小红书广告调试

# 合并后
小红书广告归因：完成归因对接与回传调试
```

聚合维度：按项目 + 按功能/模块/平台（如”小红书”、”vivo”、”OCR”、”财务”）。

### OKR/KR 归类

根据 `references/my-2026-h1-okr.md` 中的归类建议，将每条聚合后的事项映射到对应 KR。无法明确归属的放到”其他工作”分组。

归类完成后，进入”改写模式”按输出结构生成最终周报。输出中不出现项目名/仓库名等内部来源标注。

## 好周报的特点

优先体现这些特点：

- **对齐 OKR**：使用清晰标签，例如 `O1-KR4：AI 测评`；无对应 OKR 的事项必要时放到“暂无 OKR”。
- **结果前置**：写“完成开发与调试，支持能力落地”，不要只写“做了开发”。
- **能量化就量化**：尽量写清数量、比例、金额、完成状态、响应时长、准确率、日期或覆盖范围。
- **风险可见**：写清阻塞和处理方式，例如“因当前暂不具备联调条件，已先完成开发实现，待环境满足后验证闭环”。
- **过程价值清楚**：说明工作如何降低风险、提升稳定性、提升准确性、提高效率或保障上线。
- **协作有意义**：只有当协作能说明进展、依赖或推进难点时才写，不要作为填充内容。
- **后续动作明确**：结尾写具体跟进动作，不写空泛表态。
- **简洁但不单薄**：每条 bullet 都要承载一个明确贡献。

避免这些弱表达：

- 只列任务、不写结果：例如“配合测试”“完成需求”“处理问题”。
- 每条都用“配合”开头，导致贡献显得被动。
- 没有依据地写“大幅提升”“显著优化”等泛化结论。
- 隐藏阻塞。清楚写出阻塞和应对方式，比假装全部完成更可信。
- 思考部分只是重复工作清单，没有体现判断或复盘。

## 改写模式

每个事项按这个结构改写：

```text
原始事项：做了什么
优化表达：围绕【目标/场景】，完成【动作/交付物】，解决/保障/推动【结果】，当前【状态/风险/后续】。
```

常用动词：

- 推进、完成、支持、保障、打通、梳理、排查、修复、验证、沉淀、优化、跟进、闭环
- 只有用户确实是辅助角色时才用“配合”。其他情况优先用“支持完成”“协助推进”“完成验证”“保障上线”等表达。

常用结果词：

- 稳定性、准确性、可用性、兼容性、时效、效率、转化、体验、上线质量、风险前置、问题闭环

常用阻塞表达：

- "因 X 暂不具备条件，已先完成 Y，后续待 Z 后继续验证闭环。"
- "当前主要风险在于 X，已通过 Y 方式前置处理，后续重点跟进 Z。"
- "该项工作对 X 要求较高，因此本周重点放在 Y，减少后续 Z 风险。"

## 推荐输出结构

改写普通 OKR 周报时，优先使用这个结构：

```markdown
**简述近期 OKR 重要进展或规划**

本周 OKR 重点推进工作如下：

**O1-KR2：财务优化**

- ...
- ...

**O1-KR4：AI 测评**

- ...
- ...

**O1-KR3：项目迭代**

- ...
- ...

**简述近期实质性思考和分享**

本周在 ... 过程中，进一步感受到 ...

后续会 ...

同时也意识到 ...
```

如果用户所在公司的模板标题不同，沿用用户的标题。

## 思考与分享写法

思考部分建议回答下面 2-4 个问题：

- 本周哪类工作最难、风险最高？
- 用户做了什么判断或取舍？
- 从重复问题中总结出了什么规律？
- 下次哪些风险可以更早识别和处理？
- 哪些依赖还需要继续跟进，才能形成闭环？
- 用户在哪些能力上有进步：问题定位、风险识别、跨团队沟通、交付质量、数据意识、流程思考？

思考要具体，不要空泛。可参考这些表达：

- "AI 类需求更依赖外部服务稳定性和异常兜底，后续会在开发阶段提前关注超时、失败重试和模型兼容。"
- "广告归因类需求对第三方接口和联调环境依赖较强，本周先完成可开发部分，避免因单个平台阻塞影响整体进度。"
- "财务类需求对数据准确性要求较高，后续测试会继续重点关注数据流转、边界场景和流程衔接。"

## 最终检查清单

输出前检查：

- 领导能在 1 分钟内看懂重点。
- 每个 KR 都有明确进展或当前状态。
- 重要工作没有被平淡措辞削弱。
- 没有编造数据。
- 阻塞问题写了应对方式或下一步。
- 思考部分体现判断，而不只是态度。
- 整体语气专业、真实、克制，不刻意堆砌辞藻。
